# PFAS Phase Reset and Revalidation
## Pre-design audit for Sharp v1

Date:
2026-03-19

This document exists because the visual layer moved faster than the core.

That is not a disaster, but it is a signal.

If the design still feels unsatisfying, the most likely reason is not "pick a different accent color." The more likely reason is that the product truth is not broad enough yet, so the interface is being asked to look finished before the engine is finished.

This document resets the execution order.

---

# 1. Persona debate

## Persona A. Product strategist

Claim:

- The project should ship a sharp product, not a timid prototype.
- But "sharp" does not mean "visually finalized first."
- If the visual shell hardens before the engine broadens, the team will keep redesigning the same narrow product surface.

Conclusion:

- Keep the current design only as a provisional shell.
- Do not spend more cycles on final brand polish yet.

## Persona B. Domain expert

Claim:

- Public-water interpretation is partially real now.
- Private-well guidance and typed result logic both exist now, but breadth and benchmark coverage still trail the public-water path.
- The current benchmark and cost layers are credible as seeds, not yet broad enough for a full Sharp v1 claim.

Conclusion:

- The project has a credible alpha core.
- It does not yet have full decision coverage.

## Persona C. Data systems architect

Claim:

- The file-backed architecture is directionally correct.
- Source registry, benchmark registry, normalized observations, and result schema exist.
- But raw -> normalized -> derived is still mostly hand-seeded, not yet a disciplined operating pipeline.

Conclusion:

- The architecture is right.
- The operating system around the architecture is still incomplete.

## Persona D. Search strategist

Claim:

- The current engine supports a small set of high-intent pages.
- It does not yet justify scaled utility/state expansion.
- Methodology, publication metadata, review ownership, and page-level evidence blocks need to exist before search expansion becomes safe.

Conclusion:

- Search can move on a narrow front.
- Search should not scale yet.

## Persona E. Editorial director

Claim:

- The product has a point of view.
- The writing structure is stronger than the final visual identity.
- Final design should be frozen only after trust blocks, result density, and publication surfaces are stable.

Conclusion:

- Design is not blocked forever.
- Design is simply not the current bottleneck.

## Persona F. Execution skeptic

Claim:

- There is a hidden risk: the current UI can create the illusion that Sharp v1 is already complete.
- That would be a mistake.

Conclusion:

- The repo should explicitly label the current state as:
  - public-water decision alpha
  - private-well decision beta
  - final visual system deferred

# 2. Audit against the anchors

## What is genuinely in place

- file-backed source registry
- benchmark registry
- normalized public-water system seed
- normalized utility observation seed
- normalized certification claim seed
- normalized filter product and cost seed
- public-water typed decision service
- public-water typed result schema
- server-rendered home, checker, and seeded result pages
- trust-aware language pattern in the main result surface

## What exists only partially

- private-well routing exists
- private-well typed result output exists
- action checker exists
- action checker covers the core documented state space but still needs broader seeded coverage
- result schema exists across both public-water and private-well paths
- trust metadata and guide depth still lag the engine core

## What the docs require but the product does not yet fully have

- full direct-data state coverage:
  - `official_notice`
  - `indirect_data_status`
  - `current_filter_status`
- trust publication layer:
  - methodology page
  - editorial owner and methodology owner presentation without fake reviewer claims
  - last verified presentation on all public pages
  - source-ledger depth on guide pages
- broader data coverage:
  - more state guidance
  - more public-water systems
  - more certification listings
  - more cost observations
- operational validation:
  - stronger file validation
  - freshness checks
  - manual review flags across more paths

---

# 3. Hard conclusion

The project should not treat visual design as the next hard problem.

The next hard problem is breadth and rigor of the engine.

In plain terms:

- the architecture is good enough to continue
- the product definition is still good
- the current order needs correction

The corrected order is:

1. data breadth
2. decision coverage
3. publication trust layer
4. high-intent entry pages
5. final design system
6. controlled SEO expansion

---

# 4. Phase reset

## Phase A. Data breadth first

Required:

- expand state guidance beyond the current seed set
- expand public-water system coverage beyond one system
- expand certification claims beyond a single product path
- expand cost observations with confidence notes

Exit condition:

- the engine has enough breadth that the UI is rendering a real product surface, not a single showcased case

## Phase B. Decision coverage second

Required:

- broaden private-well benchmark and analyte coverage
- expand public-water utility breadth with official-source dossiers
- keep closing the gap between the documented decision table and the actual routing logic

Exit condition:

- both major household paths have typed outputs, not just guidance pages

## Phase C. Trust publication layer third

Required:

- methodology page
- source policy page
- publication metadata on public pages
- last verified and ownership presentation
- guide pages with source ledger depth and non-fiction trust blocks

Exit condition:

- the product is visibly trustworthy even before a user opens raw JSON

## Phase D. Entry-page cluster fourth

Required:

- the six manually authored high-intent pages already approved in the search anchor
- strong links from each page into the engine

Exit condition:

- search visitors can land on a page and move into the engine without confusion

## Phase E. Final visual system fifth

Required:

- re-open design only after the previous phases stabilize
- use actual content density, trust blocks, and result structures as the design material

Exit condition:

- the final interface feels specific because the product is specific

---

# 5. What to do now

The next implementation order should be:

1. expand source and normalized seed coverage
2. deepen guide content and source-ledger presentation
3. align action checker inputs with the real decision table as breadth increases
4. harden methodology and publication metadata surfaces
5. keep expanding the high-intent content cluster with file-backed guides
6. return to final design only after the above is real

---

# 6. What not to do now

- do not spend another iteration only on surface polish
- do not declare the current UI final
- do not start scaled SEO
- do not widen product comparison coverage before evidence coverage widens

---

# 7. One-sentence reset

> The current UI is a provisional shell over a promising engine alpha; Sharp v1 now has both public-water and private-well typed results, but it still requires broader data, deeper guides, and a more explicit trust layer before final design is locked.
