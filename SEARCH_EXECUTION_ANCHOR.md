# 1. 여러 페르소나 토론

## Persona A. Founder / Growth Lead

주장:

- 시장은 빨리 선점해야 한다.
- 너무 조심하면 결국 아무 페이지도 못 만들고 기회가 사라진다.
- head term을 피하더라도 mid-tail decision intent는 빨리 점유해야 한다.

원하는 것:

- 빠른 출시
- 빠른 페이지 생산
- 빠른 내부 링크 구조

## Persona B. Search Quality Skeptic

반론:

- PFAS와 음용수는 YMYL 압력이 강한 주제다.
- utility/state/helper 페이지를 성급하게 늘리면 scaled/thin으로 보일 수 있다.
- source discipline과 evidence policy 없이 확장하면 나중에 전체 신뢰를 잃는다.

원하는 것:

- trust layer 선행
- source policy 선행
- no mass rollout without proof

## Persona C. Product Manager

조정:

- 느리면 안 되지만, 순서를 잘못 잡으면 나중에 다시 뜯어야 한다.
- 따라서 `표면은 빠르게`, `핵심 로직은 구조화 후`가 맞다.
- 지금 바로 해야 할 것은 shell, routing, result format, core pages다.

원하는 것:

- 설계된 속도
- 재작업이 적은 순서
- 확장 가능한 정보 구조

## Persona D. Content Strategist

주장:

- 검색을 잡으려면 페이지 수보다 page purpose가 중요하다.
- 각 페이지가 사용자의 다음 행동을 한 단계라도 더 명확하게 해야 한다.
- 단순 정보 요약, generic vendor copy, source paraphrase는 가치가 없다.

원하는 것:

- page-level unique value
- decision-intent coverage
- high editorial clarity

## Persona E. Data / Operations Lead

현실 점검:

- utility 문서, state guidance, certification claim, replacement cost는 시간이 지나면 바뀐다.
- 따라서 “많이 만들 수 있느냐”보다 “검증 가능한 업데이트 체계가 있느냐”가 중요하다.
- source freshness를 감당 못 하면 빠른 확장은 독이 된다.

원하는 것:

- update cost visibility
- source registry
- review cadence

## Persona F. Monetization Lead

유혹:

- best filter, whole-house, comparison matrix가 돈은 빨라 보인다.

반론:

- 이 프로젝트는 trust를 깎아 먹는 순간 차별점이 사라진다.
- 수익화는 가능하지만 `판단 이전의 상업 압박`은 금지다.

원하는 것:

- trust-first conversion
- high-intent pages
- low-pressure product mapping

## 토론 결론

이 프로젝트의 운영 자세는 아래 한 문장으로 고정한다.

> 우리는 빠르게 출시하되, 무리하게 확장하지 않는다. 공격적으로 진입하고, 보수적으로 단정하며, 증명된 것만 확장한다.

# 3. 어떤 SEO는 바로 해도 되고, 어떤 SEO는 아직 안 된다

## 3.1 지금 바로 해도 되는 SEO

아래는 `manual high-intent pages`로 바로 들어가도 된다.

- public water vs private well
- test first vs filter first
- how to read a CCR / utility report
- NSF 53 vs 58 for PFAS
- under-sink vs whole-house
- PFAS filter annual cost

이 페이지들은 이유가 분명하다.

- 명확한 사용자 질문이 있다.
- 대량 생성 없이도 고유 가치가 있다.
- 엔진과 자연스럽게 연결된다.
- YMYL 리스크를 과장 없이 다룰 수 있다.

## 3.2 조건부로 해야 하는 SEO

아래는 구조가 먼저 잡혀야 한다.

- utility helper pages
- state-specific private well pages
- product comparison pages
- certification-driven product matrices

조건은 아래다.

1. source layer가 명확할 것
2. page template가 실제로 고유한 next action을 만들 것
3. update cadence를 감당할 수 있을 것
4. page owner 또는 review process가 있을 것

## 3.3 지금 하면 위험한 SEO

- 전국 단위 mass utility pages
- ZIP 기반 landing page 확장
- generic “best PFAS filter” 대량 생산
- thin product roundup
- vendor spec paraphrase pages

# 4. YMYL 최소 방어가 아니라 구조적 방어

YMYL 대응은 문구 몇 줄로 끝나지 않는다.

문구는 필요하지만 충분하지 않다.

## 반드시 갖출 것

- byline 또는 editorial ownership
- methodology / source policy
- last verified date
- can say / cannot say 구분
- why this / limits block
- source tier separation

## 문구 규칙

- 진단형 단정 금지
- 노출 확정 금지
- “best” 남발 금지
- “official data currently available” 같은 한계 표현 허용
- 부족한 데이터는 결과 상태로 승격

## 운영 규칙

- 문구를 조심하는 것만으로 YMYL 대응이 끝났다고 간주하지 않는다.
- 신뢰는 `출처`, `구조`, `검토`, `표현`이 같이 만들어야 한다.

# 5. Thin / Scaled 리스크에 대한 현실적 기준

데이터를 많이 모으는 것 자체는 가치가 아니다.

페이지가 아래 중 하나라도 못 하면 thin 가능성이 높다.

1. 사용자가 다음 행동을 더 빨리 정할 수 있게 해준다.
2. 사용자가 공식 문서를 더 쉽게 찾거나 읽게 해준다.
3. 사용자가 비용/인증/유지관리를 더 정확히 비교하게 해준다.
4. 사용자가 과잉지출을 피하게 해준다.

## scaled page 발행 전 체크

모든 utility/state/helper 페이지는 아래 6개를 충족해야 한다.

1. 고유 source set이 있는가
2. 고유 next action이 있는가
3. 고유 caveat가 있는가
4. 고유 local/state context가 있는가
5. 마지막 검증일을 붙일 수 있는가
6. 이 페이지가 없으면 사용자가 놓칠 결정이 실제로 있는가

하나라도 `아니오`면 발행하지 않는다.

# 6. 속도를 죽이지 않는 확장 규칙

우리는 확장을 막는 게 아니라, `증명된 템플릿만` 확장한다.

## rule 1. manual before programmatic

어떤 페이지군이든 먼저 수작업 3개 이상을 만든다.

검증할 것:

- 정말 고유 가치가 생기는가
- 진짜 source handling이 가능한가
- 템플릿이 얇아지지 않는가

## rule 2. engine before coverage

엔진이 약하면 coverage를 늘릴수록 품질이 희석된다.

따라서 coverage expansion은 아래 이후에만 허용한다.

- action checker shell 완성
- result card schema 확정
- decision table v1 존재
- methodology page 존재

## rule 3. unique utility/page purpose

각 helper page는 최소 하나의 독립 목적이 있어야 한다.

예:

- latest CCR 찾기
- PFAS notice 확인 경로
- 보고서 읽는 법
- 데이터 없음일 때 다음 단계
- 주별 well testing 연결

## rule 4. no page just because a keyword exists

검색량이 있어도 아래가 없으면 페이지를 만들지 않는다.

- unique data
- unique interpretation
- unique action value
